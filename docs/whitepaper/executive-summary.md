# Policy-as-Code: Governing AI-Assisted Engineering at Enterprise Scale

## Executive Summary

---

AI coding assistants are transforming software engineering. DORA's 2025 State of DevOps report found that 90% of engineering organizations now use AI in development. Teams with high AI adoption complete 21% more tasks, merge 98% more pull requests, and reduce cycle time by 24%. The productivity gains are real.

But DORA's most important finding was this: **AI is an amplifier, not a fix.** It magnifies existing strengths and existing weaknesses equally. Organizations with strong engineering practices see AI accelerate their best habits. Organizations without governance see AI accelerate their worst.

Deloitte's 2026 Technology Trends report confirms the gap: only 1 in 5 enterprises have mature governance frameworks for AI agents. The remaining 80% are deploying AI assistants with limited guardrails, risking inconsistent output quality, security vulnerabilities, and compliance failures.

### The Problem

Tech services companies face a unique challenge. We deploy AI-assisted engineering across multiple client environments, each with different:
- Cloud platforms (AWS, GCP, Azure, Databricks, Snowflake)
- Enterprise architecture patterns and technology constraints
- Security and compliance requirements (HIPAA, SOC 2, GDPR, PCI-DSS)
- Development workflows and tooling (Jira, Confluence, CI/CD)
- Data governance policies and quality standards

Without a governance framework, every engagement reinvents its AI guardrails from scratch — or worse, operates without them.

### Our Approach: Policy-as-Code

We built **cc-framework**, a four-layer policy-as-code framework that embeds engineering standards, security policies, and SDLC best practices directly into the AI assistant's configuration:

1. **Core Layer** (universal) — Coding standards, security rules, testing requirements, and SDLC workflows that apply to every engineer on every project. Enforced via managed settings that cannot be overridden.

2. **Domain Layer** (discipline-specific) — Standards for data engineering, analytics/BI, ML/DS, and full-stack development. Opt-in modules that add domain expertise.

3. **Platform Layer** (cloud-specific) — Patterns for GCP, AWS, Azure, Databricks, and Snowflake. Platform-native-first principles enforced per engagement.

4. **Client Layer** (engagement-specific) — Client EA patterns, security policies, tool configurations, and governance requirements. Auto-generated by an agentic onboarding workflow.

### Key Innovation: Agentic Onboarding

The framework's cornerstone is `/client-onboard` — an agentic, human-in-the-loop workflow that:
- Reads client EA documentation, security policies, and development standards
- Maps the client's cloud platform structure and technology constraints
- Auto-generates engagement-specific configuration (managed settings, rules, MCP connections)
- Presents proposals for lead engineer review and approval
- Produces a version-controlled audit trail

This replaces the manual, error-prone process of reading documentation and writing configuration from memory.

### SDLC Coverage

The framework provides orchestrator skills for every SDLC phase: Discovery (green-field and brown-field), Design (EA-constrained architecture proposals), Build (standards-enforced implementation), Test (six-category comprehensive testing), and Deploy (PR creation, CI/CD, documentation updates). Each phase produces standardized artifacts consumed by the next.
