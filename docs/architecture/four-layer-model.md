# Architecture: The Four-Layer Model

## Overview

cc-framework uses a four-layer architecture that maps directly to Claude Code's configuration
hierarchy. Each layer has a specific scope, override behavior, and deployment mechanism.

```
┌─────────────────────────────────────────────────────────┐
│  Layer 4: CLIENT / ENGAGEMENT                           │
│  Per-project .claude/ directory                         │
│  Generated by /client-onboard, unique per client        │
├─────────────────────────────────────────────────────────┤
│  Layer 3: PLATFORM                                      │
│  Installable modules (opt-in per engagement)            │
│  AWS / Azure / GCP / Databricks / Snowflake             │
├─────────────────────────────────────────────────────────┤
│  Layer 2: DOMAIN                                        │
│  Specialization layers (opt-in per engagement)          │
│  Data engineering, analytics, ML/DS, full-stack data    │
├─────────────────────────────────────────────────────────┤
│  Layer 1: CORE (universal, immutable per client)        │
│  Our engineering standards — never changes per client   │
│  Coding standards, SDLC orchestrators, security rules   │
└─────────────────────────────────────────────────────────┘
```

## Layer Details

### Layer 1: Core

**Scope:** Universal to all engineers, all projects, all clients.

**What it contains:**
- `CLAUDE.md` — Global engineering behavior and standards
- `CODING_STANDARDS.md` — Authoritative coding standards
- `settings.json` — Default permissions
- SDLC phase orchestrator skills (`/discovery`, `/design`, `/build`, `/test`, `/deploy`)
- Engagement onboarding skill (`/client-onboard`)
- Utility skills (`/format-and-lint`, `/test-runner`)
- Universal agents (framework-verifier, test-writer, solution-architect, etc.)
- Security, git-workflow, and code-review rules

**Override behavior:** Cannot be overridden by client configuration. Layer 1 represents
our engineering standards and quality baseline. If a client's standards conflict with
Layer 1, the `/client-onboard` skill flags the conflict and proposes a compliant alternative.

**Deployed as:**
- `managed-settings.json` → system directory (enforced by Claude Code Enterprise)
- Skills, agents, rules, tools → `~/.claude/` (user global)

### Layer 2: Domain

**Scope:** Specific to a practice area (data engineering, analytics, ML/DS, full-stack data).

**What it contains:**
- Domain-specific CLAUDE.md overlay (installed as a rule file)
- Specialized skills (data-quality-check, schema-validator, pipeline-scaffolder, etc.)
- Specialized agents (schema-designer, pipeline-architect, data-modeler, etc.)
- Domain-specific rules (SQL standards, pipeline patterns, etc.)

**Override behavior:** Extends Layer 1 with domain-specific standards. Does not override
core principles. Selected per engagement based on project type.

**Deployed as:** Skills, agents, rules → `~/.claude/` (user global, installed per engagement)

**Available modules:**
| Module | Focus | Key Components |
|--------|-------|----------------|
| `data-engineering` | Pipelines, schemas, data quality | schema-designer, sql-reviewer, pipeline-scaffolder |
| `analytics` | BI, dashboards, metrics | metric-definer, dashboard-reviewer |
| `ml-ds` | ML models, experiments, features | model-validator, ml-pipeline-designer |
| `full-stack-data` | Data apps, APIs, UIs | api-integrator, api-scaffolder |

### Layer 3: Platform

**Scope:** Specific to a client's cloud platform.

**What it contains:**
- Platform-specific CLAUDE.md overlay (best practices, service patterns)
- MCP server configuration templates (connects to cloud APIs)
- Platform-specific skills (if applicable)
- Platform-specific rules (Well-Architected Framework patterns)

**Override behavior:** Extends Layers 1+2 with platform-specific patterns. Does not
override core or domain standards. Selected per engagement based on client's cloud.

**Deployed as:**
- CLAUDE.md overlay → `~/.claude/rules/` (installed per engagement)
- MCP config → project `.mcp.json` (configured with client credentials)
- Skills, rules → `~/.claude/` (installed per engagement)

**Available modules:**
| Module | MCP Ecosystem | Status |
|--------|---------------|--------|
| `gcp` | Community servers, documented patterns | MVP |
| `aws` | 14+ official AWS MCP servers | Planned |
| `azure` | 13+ official Azure MCP servers | Planned |
| `databricks` | Managed MCP with Unity Catalog | Planned |
| `snowflake` | Managed MCP with Cortex | Planned |

### Layer 4: Client / Engagement

**Scope:** Specific to one client environment. Generated by `/client-onboard`.

**What it contains:**
- `managed-settings.json` — Client-specific security and compliance policies
- `CLAUDE.md` overlay — Client EA patterns, naming conventions, approved tech
- `rules/` — Client-specific rules derived from EA documentation
- `.mcp.json` — Jira, Confluence, and cloud platform MCP configurations
- `engagement-config.md` — Audit trail of all configuration decisions

**Override behavior:** Most specific layer. Adds client-specific constraints on top of
Core + Domain + Platform. Cannot weaken security rules from Layer 1.

**Deployed as:**
- `managed-settings.json` → system directory (if using Claude Code Enterprise managed settings)
- CLAUDE.md overlay → project `.claude/CLAUDE.md`
- Rules → project `.claude/rules/`
- MCP config → project `.mcp.json`

## How Layers Compose

For a given engagement, layers are selected and composed:

**Example: Financial services client on Azure with Databricks**
```
Layer 1: Core (always)
  + Layer 2: data-engineering
  + Layer 2: analytics
  + Layer 3: azure
  + Layer 3: databricks
  + Layer 4: client-specific (generated by /client-onboard)
```

**Example: Retail client on GCP with Snowflake**
```
Layer 1: Core (always)
  + Layer 2: data-engineering
  + Layer 3: gcp
  + Layer 3: snowflake
  + Layer 4: client-specific (generated by /client-onboard)
```

## Configuration Precedence

Claude Code resolves configuration in this order (highest priority first):

1. **Managed settings** (system directory) — Layer 1 security + Layer 4 client policies
2. **Project `.claude/settings.json`** — Layer 4 project-specific settings
3. **Project `.claude/CLAUDE.md`** — Layer 4 client instructions
4. **Project `.claude/rules/*.md`** — Layers 2-4 rules
5. **User `~/.claude/settings.json`** — Layer 1 default permissions
6. **User `~/.claude/CLAUDE.md`** — Layer 1 global standards
7. **User `~/.claude/rules/*.md`** — Layers 1-3 rules

Array settings (permissions, rules) merge across scopes. Deny rules always take precedence.

## Key Design Decision: Why Four Layers?

**Alternative considered:** Flat structure (everything in one directory)
- Rejected because: different clients need different combinations, and a flat structure
  makes it impossible to selectively install components.

**Alternative considered:** Two layers (core + everything else)
- Rejected because: domain knowledge (data engineering patterns) and platform knowledge
  (GCP vs AWS patterns) are orthogonal concerns that should be independently selectable.

**Why four is right:**
- Core = universal, non-negotiable
- Domain = practice area expertise (orthogonal to platform)
- Platform = cloud-specific knowledge (orthogonal to domain)
- Client = engagement-specific (combines domain + platform + client EA)

This gives us maximum composability with minimum complexity.
